# Phase 1 release guide: `0.1.0rc1` to `0.1.0`

Status: **guide ready; release not yet approved or performed**

This is the maintainer's single, ordered runbook for completing Phase 1 and releasing Lumis SDK
`0.1.0`. Phase 1 is a capability milestone. It does **not** mean package version `1.0.0`, and it
does not make a Semantic Versioning `1.x` compatibility promise. The package sequence is:

```text
0.0.8 -> 0.1.0rc1 -> 0.1.0
```

A future decision after Sprint 7 may consider the path toward `1.0.0`. That decision is outside
this guide.

## The rule for declaring Phase 1 complete

Phase 1 is ready for final `0.1.0` only when every checkbox in the final sign-off section is
checked and linked to evidence. A red CI job, failed clean install, unresolved critical/high
security finding, ambiguous execution authority, missing RC rehearsal, or missing independent
feedback is a stop condition. Do not publish final `0.1.0` and promise to collect the evidence
later.

## Step 1 — Confirm accounts, permissions, and local tools

1. Sign in to GitHub as a maintainer with access to `soloshun/lumis-sdk`, its private project,
   Actions, environments, and releases.
2. Open the [release workflow](https://github.com/soloshun/lumis-sdk/actions/workflows/release.yml).
   Confirm the **Run workflow** button is visible. If it is not visible, stop and correct GitHub
   Actions/repository permissions.
3. Sign in to [PyPI](https://pypi.org/) and [TestPyPI](https://test.pypi.org/). Confirm that the
   `lumis-sdk` project is visible under your account and that the GitHub `pypi` and `testpypi`
   environments still use trusted publishing.
4. In a terminal, clone or open the repository and verify the tools:

   ```bash
   git switch dev
   git pull --ff-only origin dev
   gh auth status
   uv --version
   git status --short --branch
   ```

Expected result: GitHub authentication succeeds, `uv` prints a version, the branch is `dev`, and
the working tree contains no uncommitted files. Do not continue from a dirty or stale checkout.

## Step 2 — Review the Phase 1 scope and board

1. Open [epic #9](https://github.com/soloshun/lumis-sdk/issues/9) and
   [issue #57](https://github.com/soloshun/lumis-sdk/issues/57).
2. Open the [project board](https://github.com/users/soloshun/projects/2) and filter **Target
   release** to `v0.1.0`.
3. Confirm issues #54, #55, and #56 are Done. Keep #57 In Progress until the RC evidence and final
   release decision are recorded.
4. Review all open issues. An issue may remain open when it belongs to Sprint 7 or a later phase.
   Every issue required for Phase 1 or `0.1.0` must be fixed, explicitly deferred in #9 with a
   reason, or accepted as a bounded residual risk by the lead maintainer.
5. Read [the Phase 1 roadmap](../roadmap/phase-1-trustworthy-python-foundation.md). Confirm S6-01
   through S6-10 remain checked. S6-11 and S6-12 stay unchecked until this guide reaches their
   corresponding steps.

Expected result: the board, issues, and roadmap all call the target `v0.1.0`; no Phase 1 blocker
is hidden under an unrelated milestone.

## Step 3 — Arrange the independent checks early

These checks involve other people, so arrange them before building the RC.

### Independent security review

Choose a reviewer who did not author the relevant implementation. Give the reviewer:

- the exact RC commit or tag once available;
- [the threat model](../safety/threat-model.md);
- [the internal security review](../safety/security-review.md);
- [the public API inventory](../stability/public-api.md);
- the core package plus official PostgreSQL and HTTP evidence plugin source;
- the authority, configuration, webhook, plugin-loading, persistence, proposal, approval,
  verification, artifact, and dependency boundaries.

Ask for a report containing reviewer identity or organization, date, commit/tag, scope, methods,
findings with severity, out-of-scope areas, and disposition. A public-safe report or summary must
be linkable from #57. Resolve critical/high findings before final release. For a lower finding,
record the fix or a lead-maintainer acceptance with owner and follow-up date.

### Two non-Lumis adopter reproductions

Choose two people or projects that are not validating through the hosted Lumis product. Send them
the future RC tag and [standalone adoption paths](../adoption/phase-1-standalone-paths.md).

- Adopter A should reproduce the core-only local diagnosis path.
- Adopter B should reproduce the independently packaged plugin path.

Ask each adopter to return their name/project, permission to publish the attribution, date,
operating system, Python version, installed SDK/plugin versions, exact commands, observed result,
problems, and limitations. Product-only testing, an anonymous “works for me,” or a maintainer
rerunning the example does not satisfy this gate.

Expected result at this stage: named reviewer/adopters have agreed to test. Their final results are
collected after the RC is published in Step 11.

## Step 4 — Prepare `0.1.0rc1` on a release-preparation branch

Start from current `dev`:

```bash
git switch dev
git pull --ff-only origin dev
git switch -c release/0.1.0rc1
uv version 0.1.0rc1
```

Then update `src/lumis_sdk/__init__.py` so `__version__` is also `0.1.0rc1`. Verify all three
version sources agree:

```bash
uv version --short
rg -n 'version = "0.1.0rc1"|__version__ = "0.1.0rc1"' \
  pyproject.toml uv.lock src/lumis_sdk/__init__.py
```

Expected result: the command finds the version in `pyproject.toml`, `uv.lock`, and
`src/lumis_sdk/__init__.py`.

Create `docs/releases/0.1.0rc1.md` with highlights, compatibility changes, known limitations,
upgrade instructions from `0.0.8`, official plugin compatibility, and the statement that core has
no production action executor. Move the relevant `CHANGELOG.md` Unreleased entries into a dated
`[0.1.0rc1]` section.

Do not change the independently versioned official plugins unless their package contents changed.
An existing PyPI version cannot be overwritten. If a plugin changed, choose and document its own
new version and qualify it separately with the same workflow.

## Step 5 — Generate the local RC evidence

From `release/0.1.0rc1`, run exactly:

```bash
uv sync --frozen --all-groups
uv run ruff format --check .
uv run ruff check .
uv run mypy src
uv run python scripts/generate_config_schema.py --check
uv run pytest -q
uv build --out-dir dist/core-0.1.0rc1
uv run python scripts/verify_release_artifacts.py \
  dist/core-0.1.0rc1 --distribution lumis-sdk --version 0.1.0rc1
uv run python scripts/run_phase1_benchmarks.py \
  > docs/releases/0.1.0rc1-benchmarks.json
git diff --check
```

Expected result:

- Ruff reports no formatting or lint errors.
- Mypy reports no issues.
- Every checked schema says it is current.
- Every test passes.
- `dist/core-0.1.0rc1/` contains one `lumis_sdk-0.1.0rc1` wheel and one source archive.
- Artifact verification names both archives and exits successfully.
- The benchmark JSON contains all five workloads and says observations are not production SLOs.
- `git diff --check` prints nothing.

If any command fails, stop, fix it on this branch, rerun the complete block, and record the cause
in the PR. Do not selectively omit a failing check.

## Step 6 — Merge RC preparation into `dev`

1. Commit with sign-off and push:

   ```bash
   git add pyproject.toml uv.lock src/lumis_sdk/__init__.py CHANGELOG.md docs/releases
   git commit -s -m "release: prepare Lumis SDK 0.1.0rc1"
   git push -u origin release/0.1.0rc1
   ```

2. Open a PR from `release/0.1.0rc1` to `dev`.
3. Link #57, include the local commands/results, and list every remaining external gate.
4. Wait for all required checks: Python 3.11–3.13, PostgreSQL plugin, HTTP evidence plugin,
   plugin coexistence, dependency/license review, locked dependency audit, Bandit, and secret
   scan.
5. Review the diff, resolve review comments, and merge only when every required check is green.

Expected result: `dev` contains exactly `0.1.0rc1`, release notes, and benchmark evidence. Issue
#57 remains open and In Progress.

## Step 7 — Promote the RC commit from `dev` to `main`

1. Open [the `dev` to `main` comparison](https://github.com/soloshun/lumis-sdk/compare/main...dev?expand=1).
2. Create a PR titled `release: promote Lumis SDK 0.1.0rc1`.
3. Confirm the base is `main` and compare branch is `dev`.
4. Confirm the PR contains all completed Phase 1 changes, version `0.1.0rc1`, release notes, and no
   unrelated future Sprint 7 implementation.
5. Wait for all required checks, approve the release decision, and merge.

Expected result: `main` reports `0.1.0rc1` in `pyproject.toml`, `uv.lock`, and
`lumis_sdk.__version__`. Do not publish an artifact from an unmerged feature branch.

## Step 8 — Publish and verify `0.1.0rc1` on TestPyPI

1. Open the [release workflow](https://github.com/soloshun/lumis-sdk/actions/workflows/release.yml).
2. Click **Run workflow**.
3. Select branch `main` and enter:

   ```text
   package: core
   version: 0.1.0rc1
   target: testpypi
   ```

4. Start the workflow and wait for every job to finish.

Expected green evidence includes source verification, archive-content verification, reproducible
rebuild comparison, SPDX SBOM, provenance and SBOM attestations, clean wheel/sdist installs on
Python 3.11–3.13, and `Publish core 0.1.0rc1 to TestPyPI`.

Then verify the public TestPyPI artifact locally:

```bash
./scripts/verify_published_package.sh \
  lumis-sdk 0.1.0rc1 \
  https://test.pypi.org/simple/ \
  https://pypi.org/simple/
```

Expected result: the script prints `Verification passed`, imported version `0.1.0rc1`, CLI version
`0.1.0rc1`, and passed initialization/configuration/rule checks. Also open the
[TestPyPI project](https://test.pypi.org/project/lumis-sdk/) and confirm both wheel and source
archive are visible for `0.1.0rc1`.

If trusted publishing fails, do not upload manually with a long-lived token. Correct the GitHub
environment or TestPyPI trusted-publisher configuration and rerun the workflow.

## Step 9 — Publish and verify the RC on PyPI

Only continue after Step 8 passes.

1. Run the same [release workflow](https://github.com/soloshun/lumis-sdk/actions/workflows/release.yml)
   from `main` with:

   ```text
   package: core
   version: 0.1.0rc1
   target: pypi
   ```

2. Wait for every verification, clean-install, attestation, and publish job to pass.
3. Verify the PyPI artifact:

   ```bash
   ./scripts/verify_published_package.sh lumis-sdk 0.1.0rc1
   ```

Expected result: the script prints `Verification passed`, and
[PyPI](https://pypi.org/project/lumis-sdk/) displays `0.1.0rc1`. Retain links to the successful
workflow run, artifacts, SBOM, and attestations in #57.

## Step 10 — Create the GitHub RC prerelease

1. Open [GitHub releases](https://github.com/soloshun/lumis-sdk/releases) and choose **Draft a new

2. Create tag `v0.1.0rc1` from the exact `main` commit that was published.
3. Title it `Lumis SDK 0.1.0rc1`.

5. Link the PyPI/TestPyPI pages, workflow run, SBOM, attestations, known limitations, #9, and #57.

## Step 11 — Run the RC rehearsals and collect external evidence

Give the RC tag to the reviewer and adopters arranged in Step 3.

### Upgrade rehearsal

In a temporary clean environment, install the latest final release, exercise an alpha fixture,
then install the RC and migrate it:

```bash
work_dir="$(mktemp -d)"
uv venv --python 3.11 "$work_dir/venv"
uv pip install --python "$work_dir/venv/bin/python" "lumis-sdk==0.0.8"
uv pip install --python "$work_dir/venv/bin/python" --upgrade "lumis-sdk==0.1.0rc1"
"$work_dir/venv/bin/lumis" --version
cp -R docs/releases/fixtures/0.0.8-v1alpha1 "$work_dir/project"
"$work_dir/venv/bin/lumis" config migrate \
  "$work_dir/project/rules.yml" \
  --output "$work_dir/project/rules.yml" --force
"$work_dir/venv/bin/lumis" config migrate \
  "$work_dir/project/lumis.yml" \
  --output "$work_dir/project/lumis.v1.yml"
"$work_dir/venv/bin/lumis" doctor \
  --config "$work_dir/project/lumis.v1.yml"
"$work_dir/venv/bin/lumis" rules validate \
  --config "$work_dir/project/lumis.v1.yml" --json
```

The commands copy and modify only the repository's synthetic fixture under a temporary directory;
never use a customer's production config. Expected result: CLI version `0.1.0rc1`, migration to
`lumis.dev/v1`, a clean doctor report, and JSON rule validation with `valid: true`. Repeat the
clean install/import/CLI checks on Python 3.11, 3.12, and 3.13. Record exact commands, outputs, and
any migration warning.

### Official plugin coexistence rehearsal

Build or install the declared compatible versions of core and both official plugins in one clean
environment, then run:

```bash
lumis plugins doctor --json
```

Expected result: exactly the expected official plugins appear as `compatible`; discovery imports
no plugin implementation and grants no runtime authority. Record the exact package versions.

### Feedback disposition

For every security/adopter finding, do one of the following in #57:

- link the merged fix and successful retest;
- defer a non-blocking item to a numbered issue with reason and target sprint; or
- record a bounded residual-risk acceptance by the lead maintainer.

Critical/high security findings, authority ambiguity, data-loss defects, failed upgrade paths, or
broken official plugin compatibility cannot be waived for final `0.1.0`.

When the RC rehearsal, independent review, two adopter reports, and feedback disposition are
linked, check S6-11 in the roadmap.

## Step 12 — Make the RC go/no-go decision

Add a dated comment to #57 containing:

- RC tag and commit;
- TestPyPI and PyPI verification links;
- all required GitHub Actions run links;
- SBOM and provenance/SBOM attestation links;
- local benchmark JSON;
- upgrade and official-plugin rehearsal results;
- independent security report and findings disposition;
- two attributable non-Lumis adopter reports;
- unresolved issues and accepted residual risks;
- the words **GO for 0.1.0 final preparation** or **NO-GO**, with the reason.

Use **NO-GO** if any mandatory box is missing. A no-go is a valid safety outcome: fix the issue,
publish `0.1.0rc2` using the same process, and repeat the affected evidence.

## Step 13 — Prepare final `0.1.0`

After a recorded GO decision:

```bash
git switch dev
git pull --ff-only origin dev
git switch -c release/0.1.0
uv version 0.1.0
```

Update `src/lumis_sdk/__init__.py` to `0.1.0`. Create `docs/releases/0.1.0.md`, convert the RC
changelog section into the final dated `0.1.0` entry, and include the RC feedback/fixes. Run the
entire local evidence block from Step 5 with `--version 0.1.0`.

Open and merge `release/0.1.0` into `dev`, wait for every check, then open and merge the
[`dev` to `main` release PR](https://github.com/soloshun/lumis-sdk/compare/main...dev?expand=1).
Do not merge unrelated Sprint 7 work into this final release PR.

Expected result: `main` contains final version `0.1.0`, final release notes, and all checks green.

## Step 14 — Publish final `0.1.0`

Repeat the release workflow twice from `main`:

1. `package: core`, `version: 0.1.0`, `target: testpypi`.
2. Verify with:

   ```bash
   ./scripts/verify_published_package.sh \
     lumis-sdk 0.1.0 \
     https://test.pypi.org/simple/ \
     https://pypi.org/simple/
   ```

3. `package: core`, `version: 0.1.0`, `target: pypi`.
4. Verify with:

   ```bash
   ./scripts/verify_published_package.sh lumis-sdk 0.1.0
   ```

Expected result: both workflow runs are green, both indexes display `0.1.0`, and clean installed
import/CLI versions are exactly `0.1.0`.

## Step 15 — Create the final GitHub release and close Phase 1

1. Draft a GitHub release with tag `v0.1.0` from the published `main` commit.
2. Title it `Lumis SDK 0.1.0` and use `docs/releases/0.1.0.md` as the reviewed notes.
3. Do **not** mark it as a pre-release or “latest major stable API.” It is the Phase 1 pre-1.0
   release.
4. Link PyPI, the final workflow, SBOM, attestations, upgrade guide, known limitations, and #57.
5. Check S6-12 in the Phase 1 roadmap.
6. Close #57 and #9 with the final evidence comment.
7. Move both project items to Done and confirm their Target release remains `v0.1.0`.
8. Update the roadmap current-status row to Phase 1 Complete and released as `0.1.0`.
9. Only then update consuming applications such as Lumis to the public `0.1.0` package.

## Final sign-off checklist

Final `0.1.0` is ready only when every item is checked:

- [ ] Board, issues, roadmap, changelog, and release notes all say `v0.1.0`/`0.1.0`.
- [ ] `pyproject.toml`, `uv.lock`, imported `lumis_sdk.__version__`, and CLI report `0.1.0`.
- [ ] Python 3.11, 3.12, and 3.13 quality/test/build/clean-install jobs pass.
- [ ] PostgreSQL memory, HTTP JSON evidence, and plugin coexistence jobs pass.
- [ ] Dependency/license review, locked audit, Bandit, and secret scan pass.
- [ ] Schemas are current and `v1alpha1` to `lumis.dev/v1` migration rehearsal passes.
- [ ] Wheel/sdist contents pass inspection and deterministic rebuild digests match.
- [ ] SPDX SBOM and provenance/SBOM attestations are retained and linked.
- [ ] Bounded benchmark JSON is retained; no result is presented as a production SLO.
- [ ] `0.1.0rc1` or a later RC is verified from TestPyPI and PyPI.
- [ ] Final `0.1.0` is verified from TestPyPI and PyPI.
- [ ] Official plugin versions are compatible with the final core package.
- [ ] Independent security review has no unresolved critical/high finding.
- [ ] Two attributable non-Lumis adopter reports reproduce the core and plugin paths.
- [ ] All RC feedback is fixed, explicitly deferred, or accepted as bounded residual risk.
- [ ] #57 contains the dated final GO decision and complete evidence links.
- [ ] GitHub release `v0.1.0` points to the exact published `main` commit.

When all boxes are checked, Phase 1 is complete and Lumis SDK `0.1.0` is ready. This does not
declare `1.0.0`; the project remains pre-1.0 until a later roadmap decision.
