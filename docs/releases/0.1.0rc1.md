# Lumis SDK 0.1.0rc1

Lumis SDK `0.1.0rc1` is the Phase 1 release candidate. It packages the deterministic-first,
vendor-agnostic Python foundation for independent upgrade, plugin, security, and adopter review
before final `0.1.0`.

This is a pre-release. It is not package version `1.0.0`, does not claim universal production
readiness, and does not add a production action executor.

## Highlights

- Stable `lumis.dev/v1` project, rule, diagnosis-report, plugin-manifest, playbook, and recovery
  policy documents with checked JSON Schemas.
- Deterministic, idempotent migration from every released `v1alpha1` document shape and a
  `lumis config migrate` command.
- Public API inventory plus Python, configuration, serialization, plugin, persistence, CLI,
  schema, compatibility, and deprecation policies.
- Bounded evidence collection, versioned reports, structured deterministic diagnosis,
  truth-aware memory, governed proposals, explicit approval and verification, and replay metrics.
- Optional independently packaged PostgreSQL memory and HTTP JSON evidence plugins; core retains
  no mandatory database, HTTP provider, model provider, or hosted-product dependency.
- Hostile-input coverage, authority and residual-risk review, dependency/license checks, secret
  scanning, SBOM/provenance support, reproducible-build comparison, artifact inspection, and clean
  Python 3.11–3.13 installation gates.
- Navigable API/concept/configuration/plugin/security documentation, offline cookbook smoke paths,
  Prometheus/Alertmanager and custom PostgreSQL-schema walkthroughs, and written tutorial scripts.
- Public governance, support boundaries, official/community plugin ownership, contributor
  provenance, and responsible AI-assisted contribution policy.

## Upgrade from 0.0.8

Install the candidate in a clean environment:

```bash
pip install --pre "lumis-sdk==0.1.0rc1"
lumis --version
```

Migrate a copied, source-controlled alpha project and every referenced rule document before
switching paths:

```bash
lumis config migrate rules.yml --output rules.v1.yml
lumis config migrate lumis.yml --output lumis.v1.yml
lumis doctor --config lumis.v1.yml
lumis rules validate --config lumis.v1.yml --json
```

If output filenames change, update `spec.rules.files` in the migrated project. One project cannot
mix `lumis.dev/v1` and `lumis.dev/v1alpha1` rule documents.

See the [v1 migration guide](../migrations/config-v1.md),
[compatibility policy](../stability/compatibility.md), and
[public API inventory](../stability/public-api.md) before upgrading an integration.

## Official plugin compatibility

The existing independently versioned official packages are compatible with this candidate:

```bash
pip install --pre \
  "lumis-sdk==0.1.0rc1" \
  "lumis-sdk-postgres-memory==0.1.2" \
  "lumis-sdk-http-json-evidence==0.1.2"
lumis plugins doctor --json
```

Plugin discovery reads static metadata and grants no runtime authority. Loading or composing a
plugin remains explicit and policy-bound. The PostgreSQL and HTTP plugins require declared network
and secret authority; neither executes recovery actions.

## Compatibility and known limitations

- Python 3.11, 3.12, and 3.13 are supported.
- Alpha project/rule reading remains available with a typed deprecation warning. New output and
  examples use `lumis.dev/v1`.
- Core is deterministic-first and model-optional. Optional model examples do not constitute a
  stable model-runtime contract; that work belongs to later roadmap sprints.
- SQLite is a local reference adapter. Shared PostgreSQL deployments own TLS, least-privilege
  roles, retention, backup, tenancy, and network policy.
- HTTP origin allowlisting does not replace deployment DNS, egress, endpoint-trust, token-rotation,
  or provider-specific validation controls.
- Plugin code runs in the adopter's process after explicit loading; stronger process isolation and
  package-signing policy remain future work.
- The SDK produces diagnosis, proposals, approval records, and verification records. It does not
  ship an unrestricted shell, cloud, database, or production remediation executor.
- Benchmark results are regression observations, not production SLOs or efficacy claims.
- Independent security and adopter review remain mandatory gates for final Phase 1 `0.1.0` under
  issues #57 and #9.

## Candidate verification

The release candidate is accepted only after the repository's
[Phase 1 release guide](0.1.0-phase-1-release-guide.md) records:

- complete local and GitHub checks;
- wheel/source artifact inspection and reproducible rebuild comparison;
- clean Python 3.11–3.13 installs;
- TestPyPI and PyPI installed-package verification;
- SBOM and provenance/SBOM attestations;
- upgrade and official-plugin coexistence rehearsals;
- independent security review and two attributable non-Lumis adopter reports; and
- an explicit final `0.1.0` go/no-go decision.
